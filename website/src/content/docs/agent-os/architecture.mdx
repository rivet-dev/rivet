- **Client -> Actor -> VM** architecture with clear separation of concerns
- **Secure-exec kernel** powered by WebAssembly and V8 isolates (same as Cloudflare and Chromium)
- **SQLite VFS** for filesystem persistence across sleep/wake cycles
- **Event-driven** communication from VM to actor to client

## System diagram

```
┌─────────┐     ┌──────────────────────┐     ┌──────────────────┐
│  Client  │────>│   agentOS Actor      │────>│   Secure-Exec VM │
│          │<────│                      │<────│                  │
│ (browser │     │ - Session management │     │ - WASM runtime   │
│  or      │     │ - Event broadcasting │     │ - Node.js isolate│
│  server) │     │ - SQLite persistence │     │ - Python runtime │
│          │     │ - Preview URL tokens │     │ - Virtual FS     │
└─────────┘     └──────────────────────┘     │ - Process table  │
                                              │ - Network stack  │
                                              └──────────────────┘
```

**Client** connects to the actor via WebSocket or HTTP. Calls actions, subscribes to events.

**agentOS Actor** manages the VM lifecycle, persists session data in SQLite, and broadcasts events to connected clients.

**Secure-Exec VM** is the isolated execution environment. Three runtimes mount into the kernel:
- **WASM** for POSIX utilities (coreutils, sh, grep, etc.)
- **Node.js** for sandboxed V8 isolates running agent code
- **Python** for CPython via Pyodide with kernel-backed I/O

## Event flow

Events flow from the VM through the actor to all connected clients.

```
VM Process stdout ──> Actor processOutput event ──> Client A
                                                ──> Client B
                                                ──> Client C

Agent session event ──> Actor sessionEvent ──> All clients
                    ──> onSessionEvent hook (server-side)

Permission request ──> Actor permissionRequest ──> Client / hook
                   <── respondPermission <──────── Client / hook
```

Events are:
1. **Generated** inside the VM (process output, session events, permission requests)
2. **Captured** by the actor, persisted to SQLite if applicable, and server-side hooks are called
3. **Broadcasted** to all connected clients

## Persistence

agentOS uses the actor's SQLite database for all persistent state. Filesystem contents, session records, event history, and preview tokens all survive sleep/wake cycles. Running processes and VM kernel state do not.

See [Persistence & Sleep](/docs/agent-os/persistence) for the full lifecycle, table schemas, and sleep/wake behavior.

## Core vs actor layer

The **core layer** (`@rivet-dev/agent-os-core`) manages the VM kernel directly. The **actor layer** (`rivetkit/agent-os`) wraps it with persistence, event broadcasting, sleep/wake, and orchestration.

See [agentOS Core](/docs/agent-os/core) for the standalone SDK and a detailed comparison.

## VM isolation

Each actor runs its own isolated VM using WebAssembly and V8 isolates (same sandboxing as Chromium and Cloudflare Workers), with a virtual filesystem and network stack that prevent host access.

See [Security](/docs/agent-os/security) for resource limits, network controls, and authentication.

## Performance

agentOS VMs add minimal overhead compared to native execution. WebAssembly runs at near-native speed, V8 isolates use the same engine as Node.js, and the virtual filesystem is backed by SQLite.

- **Cold boot**: VM initialization + filesystem restoration from SQLite
- **Warm boot** (wake from sleep): filesystem restoration only. Scales with persisted filesystem size.
- **Small files** (< 1 MB): negligible latency
- **Batch operations**: `writeFiles` and `readFiles` are significantly faster than individual calls

| | agentOS VM | Traditional container | Traditional VM |
|-|------------|----------------------|----------------|
| Boot time | Milliseconds | Seconds | Minutes |
| Memory overhead | Low | Medium | High |
| Isolation | WebAssembly + V8 | Namespace/cgroup | Hypervisor |
| Filesystem | SQLite VFS | Overlay FS | Disk image |
