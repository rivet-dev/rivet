Cloudflare D1 is a shared, managed SQLite service accessed via Worker bindings. Rivet provides **actor-local SQLite** where each actor instance owns its own database. The single biggest semantic difference is that D1 databases are shared across many Workers, while Rivet SQLite is scoped to one actor. You must decide whether each D1 database maps to a single actor, multiple sharded actors, or an external shared database like PostgreSQL.

## Status Summary

**Status legend:**

- **native** = direct Rivet equivalent exists
- **pattern** = achievable with a documented Rivet pattern
- **external** = requires an external service or library
- **vendor** = Cloudflare-proprietary service replaced by a recommended third-party SDK or provider
- **unsupported** = no current Rivet equivalent
- **out-of-scope** = platform-specific; not applicable to Rivet

## Database Creation and Management

In D1, databases are standalone resources created via CLI or REST API. In Rivet, each actor instance owns its own SQLite database. Creating an actor is the equivalent of creating a database.

### Before (Cloudflare)

[D1 Overview](https://developers.cloudflare.com/d1/)

```bash
npx wrangler d1 create my-database
npx wrangler d1 list
npx wrangler d1 delete my-database
```

### After (Rivet)

[SQLite](https://rivet.dev/docs/actors/sqlite)

```ts
import { actor, setup } from "rivetkit";
import { createClient } from "rivetkit/client";
import { db } from "rivetkit/db";

const myDatabase = actor({
  db: db({
    onMigrate: async (database) => {
      await database.execute(`
        CREATE TABLE IF NOT EXISTS users (
          id INTEGER PRIMARY KEY AUTOINCREMENT,
          name TEXT NOT NULL
        )
      `);
    },
  }),
  actions: {
    query: async (c, { sql }: { sql: string }) => {
      return (await c.db.execute(sql)) as Record<string, unknown>[];
    },
  },
});

const registry = setup({ use: { myDatabase } });

// Creating an actor implicitly creates its SQLite database.
// The actor key determines identity.
const client = createClient<typeof registry>("http://localhost:6420");
const dbHandle = client.myDatabase.getOrCreate(["user-123"]);

// Query the actor's database through actions
const rows = await dbHandle.query({ sql: "SELECT * FROM users" });
```

### Behavioral Delta

- D1 databases are standalone shared resources; Rivet databases are scoped to an actor instance.
- There is no `list` or `delete` equivalent. Actor lifecycle manages database lifecycle.

## Bindings and Configuration

D1 requires explicit binding declarations in `wrangler.jsonc`. Rivet actor-local SQLite needs no binding; it is available on `c.db` automatically.

### Before (Cloudflare)

[D1 Bindings](https://developers.cloudflare.com/d1/configuration/bindings/)

```json
{
  "d1_databases": [
    {
      "binding": "DB",
      "database_name": "prod-db",
      "database_id": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
    }
  ]
}
```

### After (Rivet)

[SQLite](https://rivet.dev/docs/actors/sqlite)

```ts
import { actor } from "rivetkit";
import { db } from "rivetkit/db";

// No binding configuration needed. SQLite is available
// on every actor instance via c.db.
const myActor = actor({
  db: db(),
  onRequest: async (c, request) => {
    const result = await c.db.execute("SELECT 1 as value");
    return Response.json(result);
  },
  actions: {},
});
```

## Worker Binding API (D1Database Methods)

D1 exposes `prepare`, `batch`, `exec`, `dump`, and `withSession` on the D1Database object. Rivet provides `c.db.execute(...)` for parameterized queries, transactions via `BEGIN`/`COMMIT`, and Drizzle integration. `dump()` and session bookmarks are D1-specific features tied to its shared database model and do not apply to actor-local SQLite.

### Before (Cloudflare)

[D1 Client API](https://developers.cloudflare.com/d1/worker-api/d1-database/)

```ts @nocheck
interface D1Database {
  prepare(sql: string): D1PreparedStatement;
  batch(statements: D1PreparedStatement[]): Promise<D1Result[]>;
  exec(sql: string): Promise<D1ExecResult>;
}

interface D1PreparedStatement {
  bind(...values: unknown[]): D1PreparedStatement;
  run(): Promise<D1Result>;
  raw(): Promise<unknown[][]>;
  first(column?: string): Promise<unknown>;
}

interface D1Result {
  success: boolean;
  results: unknown[];
  meta: { changes: number; rows_read: number; rows_written: number };
}

interface D1ExecResult {
  count: number;
  duration: number;
}

declare const env: { DB: D1Database };
declare const userId: string;

// prepare + bind + run
const result = await env.DB
  .prepare("SELECT * FROM users WHERE id = ?")
  .bind(userId)
  .run();

// batch (sequential, atomic rollback)
const batchResult = await env.DB.batch([
  env.DB.prepare("INSERT INTO users (name) VALUES (?)").bind("Alice"),
  env.DB.prepare("INSERT INTO users (name) VALUES (?)").bind("Bob"),
]);

// exec (raw SQL, no bindings)
const execResult = await env.DB.exec("PRAGMA table_list");
```

### After (Rivet)

[SQLite](https://rivet.dev/docs/actors/sqlite)

```ts
import { actor } from "rivetkit";
import { db } from "rivetkit/db";

declare const userId: string;

const myActor = actor({
  db: db({
    onMigrate: async (database) => {
      await database.execute(`
        CREATE TABLE IF NOT EXISTS users (
          id TEXT PRIMARY KEY,
          name TEXT NOT NULL
        )
      `);
    },
  }),
  actions: {
    demo: async (c) => {
      // Single parameterized query (params are spread, not in an array)
      const result = await c.db.execute("SELECT * FROM users WHERE id = ?", userId);

      // Batch via transaction
      await c.db.execute("BEGIN");
      await c.db.execute("INSERT INTO users (id, name) VALUES (?, ?)", "1", "Alice");
      await c.db.execute("INSERT INTO users (id, name) VALUES (?, ?)", "2", "Bob");
      await c.db.execute("COMMIT");

      // Raw SQL execution
      const tables = await c.db.execute("PRAGMA table_list");

      return { result, tables };
    },
  },
});
```

### Behavioral Delta

- `dump()` has no Rivet equivalent.
- `withSession()` and `getBookmark()` do not exist. There is no read-replica consistency model.
- Batch rollback is manual via `BEGIN`/`ROLLBACK` instead of automatic.

## Prepared Statement Methods

D1 provides `bind`, `run`, `raw`, and `first` on prepared statements. Rivet uses `c.db.execute(sql, ...params)` which fully supports parameterized queries with results returned as row objects.

### Before (Cloudflare)

[D1 Prepared Statements](https://developers.cloudflare.com/d1/worker-api/d1-database/#prepare)

```ts @nocheck
interface D1Database {
  prepare(sql: string): D1PreparedStatement;
}

interface D1PreparedStatement {
  bind(...values: unknown[]): D1PreparedStatement;
  run(): Promise<{ results: unknown[]; meta: { changes: number } }>;
  first(column?: string): Promise<unknown>;
  raw(): Promise<unknown[][]>;
}

declare const env: { DB: D1Database };

// .run() returns D1Result with metadata
const { results, meta } = await env.DB
  .prepare("SELECT * FROM customers WHERE company = ?")
  .bind("Acme")
  .run();

// .first() returns single row
const row = await env.DB
  .prepare("SELECT * FROM customers WHERE id = ?")
  .bind(11)
  .first();

// .raw() returns arrays-of-arrays
const raw = await env.DB
  .prepare("SELECT id, name FROM customers")
  .raw();
// [[11, "Alice"], [12, "Bob"]]
```

### After (Rivet)

[SQLite](https://rivet.dev/docs/actors/sqlite)

```ts
import { actor } from "rivetkit";
import { db } from "rivetkit/db";

const myActor = actor({
  db: db({
    onMigrate: async (database) => {
      await database.execute(`
        CREATE TABLE IF NOT EXISTS customers (
          id INTEGER PRIMARY KEY,
          name TEXT NOT NULL,
          company TEXT
        )
      `);
    },
  }),
  actions: {
    demo: async (c) => {
      // Returns row objects directly
      const rows = await c.db.execute(
        "SELECT * FROM customers WHERE company = ?",
        "Acme",
      );

      // First row: use LIMIT 1
      const [row] = await c.db.execute(
        "SELECT * FROM customers WHERE id = ? LIMIT 1",
        11,
      );

      // No built-in raw mode; results are always row objects.
      // Map to arrays if needed:
      const raw = rows.map((r: Record<string, unknown>) => Object.values(r));

      return { rows, row, raw };
    },
  },
});
```

### Behavioral Delta

- D1 wraps results in a metadata envelope with success, meta, and results fields; Rivet returns row arrays directly.
- No `rows_read`/`rows_written` metadata is provided.

## Return Objects

D1 returns `D1Result` (with `success`, `meta`, and `results`) and `D1ExecResult` (with `count` and `duration`). Rivet returns row arrays directly. Metadata like change counts is available via SQLite built-in functions such as `changes()`.

### Before (Cloudflare)

[D1 Query Results](https://developers.cloudflare.com/d1/worker-api/d1-result/)

```ts @nocheck
interface D1Database {
  prepare(sql: string): D1PreparedStatement;
}

interface D1PreparedStatement {
  bind(...values: unknown[]): D1PreparedStatement;
  run(): Promise<{ success: boolean; meta: { changes: number; rows_read: number } }>;
}

declare const env: { DB: D1Database };

const result = await env.DB
  .prepare("INSERT INTO users (name) VALUES (?)")
  .bind("Alice")
  .run();

console.log(result.success);        // true
console.log(result.meta.changes);   // 1
console.log(result.meta.rows_read); // 0
```

### After (Rivet)

[SQLite](https://rivet.dev/docs/actors/sqlite)

```ts
import { actor } from "rivetkit";
import { db } from "rivetkit/db";

const myActor = actor({
  db: db({
    onMigrate: async (database) => {
      await database.execute(`
        CREATE TABLE IF NOT EXISTS users (
          id INTEGER PRIMARY KEY AUTOINCREMENT,
          name TEXT NOT NULL
        )
      `);
    },
  }),
  actions: {
    insertUser: async (c) => {
      await c.db.execute("INSERT INTO users (name) VALUES (?)", "Alice");

      // Use SQLite built-in functions for metadata
      const [{ count }] = (await c.db.execute("SELECT changes() as count")) as { count: number }[];
      console.log(count); // 1
    },
  },
});
```

## Migrations

D1 uses versioned `.sql` files managed by Wrangler CLI. Rivet runs migrations inside the actor on startup via `onMigrate` or Drizzle.

### Before (Cloudflare)

[D1 Migrations](https://developers.cloudflare.com/d1/reference/migrations/)

```bash
npx wrangler d1 migrations create my-db init-schema
# Edit migrations/0000_init-schema.sql
npx wrangler d1 migrations apply my-db --local
npx wrangler d1 migrations apply my-db --remote
```

### After (Rivet)

[SQLite](https://rivet.dev/docs/actors/sqlite)

```ts
import { actor } from "rivetkit";
import { db } from "rivetkit/db";

const myActor = actor({
  db: db({
    // Migrations run automatically on actor startup
    onMigrate: async (database) => {
      // Migration 1: initial schema
      await database.execute(`
        CREATE TABLE IF NOT EXISTS customers (
          id INTEGER PRIMARY KEY,
          name TEXT NOT NULL,
          email TEXT UNIQUE
        )
      `);

      // Migration 2: add column
      // Use IF NOT EXISTS or check before altering to make idempotent
      const cols = (await database.execute("PRAGMA table_info(customers)")) as { name: string }[];
      if (!cols.some((col) => col.name === "created_at")) {
        await database.execute("ALTER TABLE customers ADD COLUMN created_at INTEGER");
      }
    },
  }),
  actions: {
    getCustomers: async (c) => {
      return (await c.db.execute("SELECT * FROM customers")) as {
        id: number;
        name: string;
        email: string;
        created_at: number | null;
      }[];
    },
  },
});
```

### Behavioral Delta

- Migrations are code, not separate `.sql` files.
- No `list` or `apply` CLI commands; migrations run automatically on actor startup.

## Import and Export

D1 provides CLI-based import/export. In Rivet, build custom admin actions or HTTP handlers for data import and export.

### Before (Cloudflare)

[D1 Overview](https://developers.cloudflare.com/d1/)

```bash
# Export
npx wrangler d1 export my-db --remote --output=./backup.sql

# Import
npx wrangler d1 execute my-db --remote --file=./data.sql
```

### After (Rivet)

[Actions](https://rivet.dev/docs/actors/actions)

```ts
import { actor } from "rivetkit";
import { db } from "rivetkit/db";

const dbActor = actor({
  db: db({
    onMigrate: async (database) => {
      await database.execute(`
        CREATE TABLE IF NOT EXISTS data (
          id INTEGER PRIMARY KEY,
          value TEXT
        )
      `);
    },
  }),

  onRequest: async (c, request) => {
    const url = new URL(request.url);

    if (url.pathname.endsWith("/export")) {
      const tables = (await c.db.execute(
        "SELECT name FROM sqlite_master WHERE type='table'",
      )) as { name: string }[];
      const dump = [];
      for (const t of tables) {
        const rows = await c.db.execute(`SELECT * FROM "${t.name}"`);
        dump.push({ table: t.name, rows });
      }
      return Response.json(dump);
    }

    if (url.pathname.endsWith("/import") && request.method === "POST") {
      const { statements } = (await request.json()) as { statements: string[] };
      await c.db.execute("BEGIN");
      for (const sql of statements) {
        await c.db.execute(sql);
      }
      await c.db.execute("COMMIT");
      return Response.json({ ok: true });
    }

    return new Response("Not Found", { status: 404 });
  },

  actions: {},
});
```

## REST API

D1 exposes HTTP endpoints for management and queries. In Rivet, build equivalent HTTP surfaces using the actor's request handler.

### Before (Cloudflare)

[D1 Overview](https://developers.cloudflare.com/d1/)

```bash
curl -X POST \
  "https://api.cloudflare.com/client/v4/accounts/{ACCOUNT_ID}/d1/database/{DB_ID}/query" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \

```

### After (Rivet)

[Actions](https://rivet.dev/docs/actors/actions)

```ts
import { actor } from "rivetkit";
import { db } from "rivetkit/db";

// Actor exposes query endpoint via onRequest
const dbActor = actor({
  db: db({
    onMigrate: async (database) => {
      await database.execute(`
        CREATE TABLE IF NOT EXISTS users (
          id INTEGER PRIMARY KEY,
          name TEXT
        )
      `);
    },
  }),

  onRequest: async (c, request) => {
    if (request.method === "POST") {
      const { sql, params } = (await request.json()) as {
        sql: string;
        params?: unknown[];
      };
      const rows = await c.db.execute(sql, ...(params || []));
      return Response.json({ results: rows });
    }
    return new Response("Method Not Allowed", { status: 405 });
  },

  actions: {},
});
```

## Retry Logic

D1 auto-retries read queries up to 2 times. Rivet retries are app-defined using workflows or queues.

### Before (Cloudflare)

[D1 Overview](https://developers.cloudflare.com/d1/)

```ts @nocheck
// D1 automatically retries read queries.
// For writes, implement manual retry:
function shouldRetry(err: unknown, attempt: number): boolean {
  const msg = String(err);
  const retryable =
    msg.includes("Network connection lost") ||
    msg.includes("storage caused object to be reset");
  return attempt <= 5 && retryable;
}
```

### After (Rivet)

[SQLite](https://rivet.dev/docs/actors/sqlite)

```ts
import { actor } from "rivetkit";
import { db } from "rivetkit/db";

// Simple retry wrapper
async function withRetry<T>(fn: () => Promise<T>, maxAttempts = 3): Promise<T> {
  for (let i = 0; i < maxAttempts; i++) {
    try {
      return await fn();
    } catch (err) {
      if (i === maxAttempts - 1) throw err;
    }
  }
  throw new Error("unreachable");
}

const myActor = actor({
  db: db({
    onMigrate: async (database) => {
      await database.execute(`
        CREATE TABLE IF NOT EXISTS users (
          id TEXT PRIMARY KEY,
          name TEXT
        )
      `);
    },
  }),
  actions: {
    getUser: async (c, { userId }: { userId: string }) => {
      // Use retry wrapper around database queries
      const rows = await withRetry(() =>
        c.db.execute("SELECT * FROM users WHERE id = ?", userId),
      );
      return rows;
    },
  },
});
```

## Horizontal Scaling Model

D1 scales out across many small databases (10 GB each). Rivet actors are a natural sharding primitive: each actor owns its own SQLite database.

### Before (Cloudflare)

[D1 Overview](https://developers.cloudflare.com/d1/)

```ts @nocheck
interface D1Database {
  prepare(sql: string): {
    bind(...values: unknown[]): {
      run(): Promise<{ results: unknown[] }>;
    };
  };
}

declare const userId: number;
declare const env: Record<string, D1Database>;

// Shard across multiple D1 databases
function getDbForUser(userId: number, env: Record<string, D1Database>): D1Database {
  const shard = userId % 4;
  return env[`DB_SHARD_${shard}`];
}

const shardDb = getDbForUser(userId, env);
const result = await shardDb
  .prepare("SELECT * FROM users WHERE id = ?")
  .bind(userId)
  .run();
```

### After (Rivet)

[Keys](https://rivet.dev/docs/actors/keys)

```ts
import { actor, setup } from "rivetkit";
import { createClient } from "rivetkit/client";
import { db } from "rivetkit/db";

declare const userId: string;
declare const regionId: string;

const userDb = actor({
  db: db({
    onMigrate: async (database) => {
      await database.execute(`
        CREATE TABLE IF NOT EXISTS profile (
          id TEXT PRIMARY KEY,
          name TEXT
        )
      `);
    },
  }),
  actions: {
    getProfile: async (c, { id }: { id: string }) => {
      return (await c.db.execute("SELECT * FROM profile WHERE id = ?", id)) as {
        id: string;
        name: string;
      }[];
    },
  },
});

const regionDb = actor({
  db: db(),
  actions: {},
});

const registry = setup({ use: { userDb, regionDb } });

// Each actor IS a shard. Routing is by actor key.
// No manual shard selection needed.
const client = createClient<typeof registry>("http://localhost:6420");
const userHandle = client.userDb.getOrCreate([`user-${userId}`]);
const result = await userHandle.getProfile({ id: userId });

// For range-based sharding, use composite keys:
const regionHandle = client.regionDb.getOrCreate([`region-${regionId}`]);
```

### Behavioral Delta

- D1 sharding requires manual database selection; Rivet actors route by key automatically.
- Cross-shard queries require application-level fan-out in both models.
- Actor-local SQLite has different size limits than D1's 10 GB cap.

Rivet actors can be deployed to any supported platform. See the [Connect docs](/docs/connect) to choose your deployment target.

## Unsupported, External, and Vendor Features

- **Shared/Cross-Actor Database** -- vendor -- D1 databases are shared across Workers. Use [PostgreSQL](https://rivet.dev/docs/actors/postgres), [Turso](https://turso.tech/), or [PlanetScale](https://planetscale.com/) for shared state across actors.
- **Time Travel (PITR)** -- unsupported -- No documented point-in-time recovery for actor-local SQLite.
- **Legacy Backups** -- unsupported -- No actor-local backup or snapshot API.
- **Read Replication** -- unsupported -- No Sessions API, bookmark consistency, or read-replica model. Redesign around actor ownership or use an external database.
- **SQL Query Execution via Wrangler** -- out-of-scope -- No D1-style SQL CLI. Use app-defined admin actions or external DB tooling.
- **Remote Development** -- out-of-scope -- No hosted playground. Use local dev with `npx rivetkit dev`.
- **Billing and Usage** -- out-of-scope -- No D1-style row metering.
- **Audit Logs** -- out-of-scope -- No actor-local audit-log surface.
- **Pricing** -- out-of-scope -- Platform/commercial comparison, not a runtime feature.
- **Wrangler Commands Reference** -- out-of-scope -- No D1-equivalent admin CLI surface.
