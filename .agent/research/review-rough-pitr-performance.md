# Performance review: sqlite-rough-pitr.md

Adversarial perf/scaling review against the rough-PITR spec extending the stateless storage architecture. All quantitative numbers below are back-of-envelope; assumes FDB at p50 ~1ms / p99 ~5ms per round-trip, S3 PUT/GET p50 ~30ms / p99 ~300ms, page size 4 KiB, SHARD_SIZE = 1024 pages = 4 MiB.

## Hot-path regressions

**1. `commit` is not 1 RTT — it is 1 RTT plus an unbudgeted META/manifest read+write (§11 step 9).** §17's table claims `commit` (steady state) = 1 RTT but step 9 of §11 says: *"`/META/manifest` read+update `last_access_ts_ms`"*. `BranchManifest` is a vbare blob covering `cold_drained_txid`, `last_hot_pass_txid`, `last_access_ts_ms`. Read-then-write is a write that takes a conflict range — every commit now has a *cross-component* conflict surface with the cold compactor and eviction sweep. Either (a) move `last_access_ts_ms` to its own atomic-set key, or (b) accept that every commit conflicts with cold's Phase C OCC region. The base spec deliberately split META into single-writer sub-keys (§"META key split"); this regresses it.

**2. `commit` adds 1 mutation for `SetVersionstampedValue` (§11 step 7, §6 `CommitRow`).** Not an RTT, but a versionstamped mutation extends commit-batch size and adds a 16-byte conflict range. FDB's `SetVersionstampedValue` is atomic with the tx commit, no extra RTT — claim of "0 RTT change" holds for this specific op. However, the resulting `versionstamp` is *not readable in the same tx* (the actual versionstamp is only known post-commit). Any caller that wants the versionstamp back from `commit` (e.g. `create_bookmark` returning a stable bookmark string) must either (a) re-read `COMMITS/{T}` after commit (1 extra RTT) or (b) use `get_versionstamp()` (FDB future, returns post-commit). The spec is silent on this. Bookmarks claim "0 (in-memory)" in §17 but the wire format uses the COMMIT's versionstamp — that requires the post-commit future, which is a real wall-clock wait equal to the commit's own latency.

**3. Read path on a fresh ActorDb is not "1 + 1" (§17). It is 1 + N for ancestry depth N.** §10 says "ancestors = [current_branch -> parent -> ... -> root]". The first read on a fresh `ActorDb` must materialize this chain before doing any PIDX work. Each ancestor requires reading `BRANCHES/list/{branch_id}` to extract `parent` and `parent_versionstamp`. These are *serial* — you cannot fetch parent N+1 until you read parent N. With `MAX_FORK_DEPTH = 16` and a (claimed but undefined) `MAX_NAMESPACE_DEPTH`, worst case is 16 serial RTTs ~16ms p50, ~80ms p99. The spec table says "1 + 1 (ancestry walk)" — that is wrong; the walk is N RTTs unless the fields are restructured to allow a range scan or the chain is stored denormalized (e.g. `ancestors_chain: Vec<(branch_id, parent_versionstamp)>` on every record).

**4. Read path on a steady-state cache hit but ancestry-not-self pgno hits 1 RTT extra per ancestor crossed (§10 step 5).** The PIDX cache is per-conn and per-`ActorDb`; the spec says "PIDX flattened-ancestry cache". But the *PIDX values* live on different physical FDB keys for each branch (`BR/{branch_id}/PIDX/...`). For a fork descendant reading a page that was never modified after fork: PIDX miss on current branch (1 RTT), PIDX miss on parent (1 RTT), ..., PIDX hit on the ancestor that owns the page (1 RTT), then SHARD/DELTA blob fetch (1 RTT). For a depth-3 actor branch chain on a depth-3 namespace branch chain (9 ancestor branches in worst-case crossover), this is 10 serial FDB RTTs for one cold page = ~10ms p50, ~50ms p99 — vs. the stateless spec's 2 RTTs for the same operation. The cache only helps if it pre-resolves *which ancestor owns each pgno*; but since pages can move between branches across rollbacks, the cache must be keyed on `(pgno, expected_owner_branch_id, expected_owner_txid)`, with no way to populate it without doing the walk first.

**5. `cold_state` check is mentioned nowhere as a hot-path read but it is implied.** §10 step 6 says cold reads "happen only after eviction has cleared the FDB resident set for this branch and the actor wakes for read." How does the read path detect "cleared"? It must observe a SHARD miss + a PIDX miss-or-stale. PIDX could be present but point at a DELTA that's been evicted; the read tx fetches the DELTA, gets nothing, and must fall through to S3. That's 1 *extra* failed FDB read per evicted page on every cold-path read. For a 100-page query against a fully-evicted branch: 100 wasted FDB reads before hitting S3.

**6. APTR `last_swapped_at_ms` cache invalidation read is on the hot path (§10).** "*detected by including a fast `APTR.last_swapped_at_ms` check in the same tx as `/META/head`*". That is one *extra* FDB key read on every `get_pages` and `commit` (now 2 keys: `/META/head` + `APTR/{actor_id}/cur`). Two parallel gets via `try_join!` is "1 RTT" only on FDB native; on RocksDB or under FDB tail latency, the second key is a real wall-clock add. This is a regression from the stateless spec's single-key steady-state read.

## Cold-path latency cliffs

**7. Eviction wake-up is fundamentally bounded by S3 GET latency × layer count (§10 step 6).** Wake from "fully evicted" to "first page served" requires:
1. APTR + BRANCHES read (1 FDB RTT).
2. Ancestry walk (N FDB RTTs as above).
3. Cold manifest GET (1 S3 GET, ~30ms p50 / ~300ms p99).
4. For each page in the query, find the layer that covers it (in-memory after manifest load).
5. GET each layer (1 S3 GET each).

For a 100 MB DB consisting of 25 image layers (4 MiB each per shard), a query touching pages across all 25 shards = 25 parallel GETs after the manifest. p50: ~1 + N + 30 + 30 ≈ ~70ms (with N=3, parallel S3 = pipelined). p99: ~5 + 5N + 300 + 300 ≈ ~620ms. For a 1 GB DB (250 image layers): query touching all shards is 250 parallel GETs. S3's per-account concurrency cap will throttle; effective wall clock is ~500ms-2s p99. Because this happens *synchronously on the first request*, the user sees a multi-hundred-ms request latency cliff.

**8. Pinned bookmark create blocks the caller for the full LTX encode + S3 PUT of the entire DB (§17 "Pinned bookmark latency").** §17 says "For a 1 GiB DB at typical S3 PUT speeds, wall-clock is seconds to tens of seconds." Quantify: 1 GiB / 50 MB/s upload = 20 seconds. Worse, the LTX encoding pass *reads every page from FDB* — for a 1 GiB DB at 4 KiB pages = 262,144 page reads. Even pipelined at FDB's max throughput, that's seconds of FDB read load before the upload starts. Each `create_pinned_bookmark` call adds substantial transient FDB read pressure. At 100 actors × 1 GiB pinning concurrently = 26M FDB reads in the same window — easily DoS's an FDB cluster. There is no rate limiter described.

**9. Cold compactor Phase B latency is unbounded relative to FDB tx age budget (§12.2).** Phase B is "S3 only, no FDB tx" so Phase B's *own* latency is fine. But Phase A snapshots SHARD versions and DELTA chunks that may be deleted by hot compactor + eviction during Phase B. Phase C OCC fence on `cold_drained_txid` aborts only if `cold_drained_txid` advanced — but Phase B's S3 PUTs may have *uploaded data that no longer corresponds to FDB state* (because hot compactor folded a shard, eviction deleted the old version). Phase B then writes orphan data to S3. The spec acknowledges idempotent PUTs but doesn't explain how stale layer entries get GC'd from cold manifest if the corresponding FDB SHARD version is gone. Operational hot spot: an unbounded-S3-latency Phase B (think 30s for a multi-GB upload) lets eviction race ahead and produce phantom layers.

## Resource scaling issues (memory, network, S3 cost, FDB cost)

**10. Versioned SHARD storage cost balloons before eviction catches up (§5.2).** For a 1 GB DB = 256 SHARDs (at SHARD_SIZE = 1024 pages = 4 MiB per shard). At 100 commits/sec, hot compactor folds DELTAs into new SHARD versions periodically. Hourly hot-compactor cadence × 256 shards × 1 hour = up to 256 * 60 = 15,360 SHARD versions if every commit touches every shard (worst case). Even at 1% shard fanout per commit (each commit touches 1 of 256 shards), 100 commits/sec × 3600s = 360K commits/hr → ~1,400 SHARD versions per shard per hour. Each is 4 MiB = 5.6 GiB *per shard*, 1.4 TiB across all shards. Eviction must run far more aggressively than "hourly" or the FDB byte multiplier explodes 1000x vs. single-SHARD. The spec's "newer SHARD version exists in FDB" eviction predicate is insufficient — it gates only on existence of a newer version and `cold_drained_txid >= max_folded_txid`, which means an evictable version remains until cold-tier upload completes. If cold upload has any backpressure, FDB grows unboundedly.

**11. Eviction global index hot spot (§12.3).** `[GLOBAL]/eviction_index/{last_access_ts}/{actor_id}` is updated on *every* read/write per the spec. At 1M actors × 100 commits/sec each = 100M FDB writes/sec just for eviction index churn. Each write also requires a *delete* of the prior `last_access_ts` row (otherwise the index becomes a write-once log). That's 200M FDB ops/sec on a single hot keyspace prefix — impossible. FDB's keyspace-balanced contention model makes a single-prefix range hot. Mitigation needed: (a) sample to 1-in-N, (b) coarsen to minute-bucket timestamps, or (c) accept tens-of-seconds staleness and update via batched per-pod flush. The spec says "lazily by `ActorDb` on access; not load-bearing for correctness" but doesn't specify the sampling rate. Without aggressive sampling this is the worst FDB hot spot in the design.

**12. PIDX storage multiplied by branch count (§6).** PIDX is per-branch (`BR/{branch_id}/PIDX/{pgno}`). Each fork creates a new branch with no PIDX entries — fork is zero-copy. But each *commit* on the forked branch creates new PIDX entries that shadow but *do not delete* the parent's PIDX entries. For an actor with 16-deep fork chain that's been long-lived, total PIDX entries = sum across all branches of pages-modified-since-fork. For a 1 GiB DB with steady churn, each branch's PIDX = ~262K entries × 12 bytes/entry ≈ 3 MB. 16 branches = 48 MB PIDX *per actor*. At 1M actors with deep histories: 48 TB of PIDX in FDB. The spec offers no PIDX consolidation pass.

**13. Cold manifest rewrite cost (§7, §12.2 Phase B step 4).** Cold manifest is rewritten on every cold-pass with appended layer entries. For a long-lived actor with thousands of layers, the manifest grows to many MB. Each pass = full manifest read + append + full manifest PUT (S3 has no append-only API for vbare blobs). At 1M actors × 1 cold pass/hour × ~1 MB manifest = 1 TB/hour of S3 PUT traffic just for manifest churn. S3 PUT at $0.005 per 1000 = $5/hour for manifest churn alone, plus bandwidth. Mitigation: split manifest into chunks or use CompleteMultipartUpload pattern, but spec doesn't.

**14. Bookmark pin storage at scale (§9).** `pin/{versionstamp_hex:32}.ltx` is a full-DB image. 100 pinned bookmarks × 1 GiB DB = 100 GiB S3 storage *per actor*. Spec doesn't say pins are ever GC'd except via explicit delete (§"§13 GC", `oldest_bookmark_pin` recompute on deletion). No max-pins-per-actor limit mentioned. At 1M actors × 100 pins each × 1 GiB = 100 PiB S3, which is a significant production cost (≈$2.3M/month at standard S3). Need a bounded-pin policy.

**15. Versionstamp + COMMITS rows accumulate without compaction (§6 `CommitRow`).** Every commit writes `COMMITS/{txid_be:8}`. For a Tier 0 actor at 100 commits/sec running for 1 year = 3.15 billion COMMITS rows × ~32 bytes each = ~100 GB FDB *per actor* in COMMITS alone. There is no COMMITS compaction. Spec §25 mentions deferred work for "Compactor for COMMITS rows" but the active design has no upper bound. Even Tier 0 (no S3) accumulates this; a long-lived high-throughput Tier 0 actor exhausts FDB quota *before* its data does.

**16. Pin counter recompute on bookmark delete is O(branches × bookmarks) (§15 "Multi-bookmark pin").** "*deletion reads bookmark and runs a per-pass full recompute (pass is rare)*". For an actor with 1000 bookmarks across 16 branches, deleting one pinned bookmark requires reading every other pinned bookmark on every branch in the actor's lineage to recompute `oldest_bookmark_pin`. With 100 pins on a 16-branch chain: 1600 reads per delete. Not a hot path but a latency cliff for pin-management UIs.

## Operational hot spots (concurrency, fanout, leader-election, throttle interaction)

**17. Hot-compactor lease per branch, not per actor (§12.1).** Spec: "*the hot compactor's lease key is `BR/{actor_branch_id}/META/compactor_lease`. Per-branch, not per-actor; an actor with N branches has N independent hot-compactor leases.*" For an actor with 16 branches all generating commits, this is 16 independent UPS triggers, 16 lease acquisitions, 16 compaction passes. Compaction throughput is divided by branch fanout. For an actor that never forks: fine, single branch. For a Tier 1+ actor with active descendants (rollback log + forks): multiplied compaction load. Throttle (`last_trigger_at`) on `ActorDb` is per-`ActorDb`, but each branch is its own thing — does the throttle apply per branch or per actor? Spec doesn't say. If per-actor, hot branches starve cold ones; if per-branch, the throttle multiplies trigger rate by branch count.

**18. Tier 0 → Tier 1 transition is a CAS race that must be observed by every fork attempt (§14).** Every `fork_*` and `create_pinned_bookmark` does `set_if_equal(tier, T0, T1)`. Concurrent first-fork on the same namespace causes one win + N-1 conflict aborts. Each abort retries, reads the tier, sees T1, proceeds. p99 fork latency under fork-storm = 2 RTTs (first attempt aborts, second succeeds). Acceptable but worth measuring.

**19. Cold compactor + eviction compactor lease contention is global (§6).** `CMPC/lease_global/{kind:1}` is a single FDB key per kind. Eviction is one global lease across all actors — that's fine for serializing eviction work, but a slow Phase A on one actor blocks eviction sweep across the entire fleet. With 1M actors, eviction must scan + decide per-actor under one lease. The spec says `BATCH_SIZE` per pass but not per-actor predicate cost. A predicate that does several FDB reads per actor × 1M actors per sweep = 1M serial FDB reads under one lease. Cycle time = ~1000 seconds (at 1ms/RTT). Eviction lag becomes operationally significant. Need parallel eviction workers under separate sub-leases (e.g. shard by `actor_id mod N`).

**20. Phase C OCC retries cascade under cold-burst load (§12.2 Phase C step 2).** "*Regular-read /META/cold_compact.cold_drained_txid; assert == the value read in Phase A. If a concurrent hot pass advanced it, abort + restart.*" Cascade scenario: hot compactor runs every 60s for an actor; cold pass (with multi-MB Phase B uploads) takes 30s. Cold's Phase A → C window is wider than hot's interval → every cold pass aborts → cold never makes progress. Hot compactor doesn't update `cold_drained_txid` per the spec — only cold does — so this concern is *partly* mitigated, but the spec lists `META/manifest.cold_drained_txid` as written by both cold (Phase C) *and* the hot compactor's `last_hot_pass_txid` mirror. Re-reading the spec carefully: hot writes `last_hot_pass_txid`, cold writes `cold_drained_txid`. They share `BranchManifest` though, which is read-modify-written by both — that's a real RMW conflict on the manifest blob unless the writes are split into separate keys.

**21. Phase B re-read of pending markers is O(active cold passes) (§12.2 Phase B step 5).** "*Re-read pending markers; clean up orphaned ones older than STALE_MARKER_AGE_MS.*" An S3 `LIST` on `pending/` returns everything; with 1M actors × occasional Tier 1+ pass, hundreds of pending markers exist concurrently. LIST is a 1000-key paginated op, billed per-call. Each cold pass paginates through all markers. Cost: 1M actors × 1 cold pass/hr × ~1 LIST = 1M LIST ops/hr = $5/hr. Mitigation: prefix markers by actor_branch_id so each pass only LISTs its own.

**22. FDB tx-age budget for `derive_branch_at` includes ancestry/pin reads (§8 primitive).** `derive_branch_at` does: `vbare_get` source record + `ensure_tier_at_least` (which walks ns ancestry?) + `atomic_min_get` bk_pin + 5 writes. If `ensure_tier_at_least` walks the namespace ancestry to root (16 deep), that's 16 serial FDB reads inside one tx. Plus the OCC fence read. Total fork tx is ~20 ops. p99 = 100ms. Within FDB's 5s budget but well above the "tens of milliseconds" claim in §17. Forking under deep namespace nesting is slow.

**23. `swapped_once: bool` field on ActorPointer makes every rollback-then-rollback-back history (§5.4).** `last_swapped_at_ms` is updated on every rollback; the APTR/history key under `APTR/{actor_id}/history/{ts_ms_be:8}` accumulates one row per rollback. No GC mentioned. A misbehaving caller doing rollback-spam grows FDB unboundedly via the history log alone.

**24. Tier 0 actors have no eviction path defined (§14, §12.3).** "*Eviction compactor: Tier 1+ only.*" Tier 0 actors never evict from FDB. Their FDB usage grows with COMMITS (issue 15) and PIDX (issue 12) without bound. The spec mentions `SqliteStorageQuotaExceeded` will reject commits past `SQLITE_MAX_STORAGE_BYTES`, but PIDX + COMMITS for an active actor easily exceeds this without touching the data quota. Need explicit Tier 0 retention semantics — either a Tier 0 COMMITS-row compaction, or auto-promote to Tier 1 at some COMMITS-row threshold. Otherwise Tier 0 is unbounded.

## Wins (places where the design IS performant)

**25. Forking is genuinely O(1) FDB metadata (§8).** No data motion at fork time; the new branch's reads fall through via parent_versionstamp. This is actually faster than Neon's similar primitive because FDB versionstamps do globally-ordered AS-OF lookups without a separate timeline coordinate.

**26. Pin computation reads three independent atomic-min counters (§13).** No tree walk for pin computation — just three reads. The atomic-min model means writers don't read during fork; the GC pass reads on its own schedule. This is genuinely scalable.

**27. Cold compactor Phase B holds no FDB tx (§12.2).** Cleanly separates S3 latency from FDB tx-age budget. The OCC fence in Phase C is the right level of conflict detection.

**28. PIDX deletes via `COMPARE_AND_CLEAR` (inherited from stateless spec, kept).** No conflict range on stale-entry deletes; commit-vs-compaction races are no-op on stale entries. Carries forward correctly to the per-branch PIDX layout.

**29. Eviction predicate's three independent gates (§12.3).** `desc_pin`, `bk_pin`, and hot-cache-window are three independent atomic-min reads — no scan, no graph walk. Predicate evaluation is O(1) per actor branch.

**30. Versionstamps as the AS-OF coordinate (§5, §6).** FDB versionstamps are ~free at commit time (one mutation, no extra RTT). Using them as the cross-branch time coordinate avoids any centralized counter; correctness is automatic. This is the right primitive.

## Summary of cliff/regression count

- Hot-path RTT regressions identified: 6 (issues 1–6).
- Cold-path latency cliffs: 3 (issues 7–9).
- Resource scaling issues: 7 (issues 10–16).
- Operational hot spots: 8 (issues 17–24).
- Wins: 6 (issues 25–30).

The "0 RTT change" claim in §17 does not hold under careful audit. The minimum hot-path regression is 1 extra read (`APTR.last_swapped_at_ms`) plus a manifest read-modify-write per commit. The realistic ancestry-walk regression is N RTTs deep. The biggest scaling concerns are the eviction global index hot spot (issue 11), versioned SHARD storage multiplier (issue 10), and unbounded COMMITS rows on Tier 0 (issue 15).
