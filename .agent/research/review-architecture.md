# Architecture review: sqlite-pitr-fork.md

Reviewed against `sqlite-storage-stateless.md`, the five research reports, and `CLAUDE.md`/`engine/CLAUDE.md`.

## Critical issues

1. **Read fall-through has no termination bound and the spec ignores its own grandparent case.** Section "Read with parent fall-through" (lines 366-373) describes a 4-step lookup: own DELTA -> own SHARD -> parent DELTA/SHARD -> cold tier. Step 3 says "fall through to parent." There is no description of what happens when the parent is itself a fork descendant whose own pages live in *its* parent. Concretely: a chain `head -> A -> B -> C` reading a pgno that has not been written since `C` requires four PIDX lookups (per branch) plus possibly four cold-tier fetches. A 10-deep chain (Turso allows 100, the spec is silent) costs 10x the steady-state RTTs. The spec markets `get_pages` as "1 RTT (warm cache, fork descendant, page NOT touched in branch)" with "+0 if parent PIDX cached, +1 if cold parent" (line 510). That table only models depth=1. **Recommended fix:** specify a max chain depth (Turso uses 100; Neon collapses on compaction), specify the fall-through algorithm explicitly as a loop walking `parent.parent.parent...`, and either require the cold compactor to materialize image layers in deep descendants to bound depth, or set a hard depth cap that `fork()` rejects. Also fix the latency table to reflect O(depth) RTTs in the worst case.

2. **GC `retention_pin_txid` algorithm cannot survive a parent-side restore-to-bookmark while a child branch is alive.** Section "Garbage collection" (lines 454-463) computes `pin = max(0, head_txid - txid_for_window(NOW - PITR_WINDOW_MS))` then `pin = min(pin, C.parent_txid)` for each child. But `restore_to_bookmark` *also* doesn't delete the original branch (line 170: "the original branch is **not deleted by restore** — it is left as a sibling"). Concretely: parent at txid 1500, fork at 800, restore parent to bookmark at 1200 (creating a new branch as the live head; old parent now a sibling). The "parent" branch the child pins is now `Frozen` and idle; its `head_txid` never advances; the 30d window math `head_txid - txid_for_window(...)` will rapidly drag the pin forward and GC the layers below 800, breaking the child. The pin should be derived from `created_at_ms`, not `head_txid - window`, for `Frozen` branches. **Recommended fix:** Frozen-branch retention is "delete iff (deleted OR (no live descendants AND created_at_ms older than retention))." Spell this out and add a test case for it.

3. **The `retention_pin_txid_minimum` monotonicity rule (line 458) makes `delete_branch` ineffective at reclaiming space.** "never advance the pin past what was already set in `META/manifest.retention_pin_txid` (monotonic)." Concretely: parent had a child at parent_txid=800 last pass, so `pin=800`. The child is then deleted. Next GC pass sees no children, so the inner `min(pin, C.parent_txid)` produces `pin = head_txid - window`. But the monotonic rule clamps pin back to 800. Forever. **The pin can never go up.** Either the rule means "never *retreat*" (never make pin *smaller*, releasing already-pinned layers) and the wording "advance" is backwards, or the rule literally pins layers forever after any child fork. Section 5 of the charter calls out Neon issue #707 — this is the same class of bug. **Recommended fix:** rewrite the predicate. The actual invariant should be: pin is the *minimum* of (window-driven floor) and (oldest live descendant's parent_txid). Monotonicity should be relative to live descendants, not historical state. The "monotonic" word is also imprecise: it is fine for the pin to *retreat* (release older layers) when a descendant goes away.

4. **Bookmark format collides with itself across siblings of the same restore tree.** Section "Bookmarks" (lines 76-86): `{timestamp_ms}-{txid}-{branch_id}`. After `restore_to_bookmark`, the new branch's first commit is at txid=parent_txid+1 with a fresh wall-clock. Two distinct branches forked from the same point both get txids starting from the same parent_txid, and their wall-clock streams can overlap arbitrarily. The lex sort across branches becomes "by wall clock then arbitrary branch_id," which **silently violates the CF DO contract** that "lex order = chronological order" within an actor's history (which the CF DO doc actually means within a single linear DO log, not a forked graph). The spec acknowledges this in open questions (line 647) but waves it off — "fine." It is not fine when `get_bookmark_for_time(t)` is supposed to return "the smallest bookmark whose timestamp_ms >= t" (line 99-100): with siblings, "smallest" is now ambiguous and depends on UUID sort order. **Recommended fix:** `get_bookmark_for_time` must take a branch context and only walk that branch's chain. Document explicitly that lex order is only meaningful within a single branch's parent chain, not across forks. Also reject the open-question hand-wave — concrete UX behavior matters here.

5. **Compactor coupling: hot vs cold both write `DELTA/*` keys, and the "tx-disjoint at any single instant" claim depends on the hot compactor honoring `materialized_txid` at FDB tx commit time, not plan time.** Section "Hot compactor and cold compactor on the same branch" (lines 540-549). Hot compactor folds DELTAs into SHARDs and updates `META/compact.materialized_txid`. Cold compactor reads `materialized_txid` (snapshot read, line 548) and clears DELTAs `<= materialized_txid`. **Race:** hot compactor reads `materialized_txid=100` (snapshot), starts folding deltas 101-132 into shards, writes `materialized_txid=132`. Concurrently cold compactor (snapshot-read) sees `materialized_txid=100` from before the hot pass, drains and clears `DELTA/{1..100}`. Cold's clear happens after hot's set; the FDB conflict-range model with snapshot reads means cold's tx never sees hot's write and never aborts. This is fine *if* the cold compactor only ever clears strictly *older* deltas than what hot's write set touched. Hot compactor doesn't touch `DELTA/{1..100}` once it's folded them, so this works — **but only because the hot compactor never re-folds**. If hot ever re-runs against an old delta range (say after a crash, or to recompact under a smaller shard size), the disjoint claim fails. **Recommended fix:** make the rule explicit and assert it: "hot compactor MUST NOT modify `DELTA/{T}` for any T at-or-below the highest `materialized_txid` ever written." Add a debug assert. Better: cold takes a regular read of `materialized_txid` (taking conflict range) so its clears observe the latest hot write.

6. **`fork at bookmark whose layers were just GC'd` resolution is hand-waved and breaks the moment a caller does `fork(Bookmark)` from a non-cached client.** Section "Failure modes" (line 580): "Atomic-add refcount happens BEFORE GC pin advance, so this is a tx ordering bug. Margin in GC pin (`GC_FORK_MARGIN_TXIDS`) makes the window safe; if margin is exceeded, return `ForkOutOfRetention`." Concrete sequence: caller resolves bookmark (reads `COMMITS/{T}` outside any fork tx), GC runs and advances pin past T and deletes the relevant layer, then caller calls `fork(Position{...,T})` which atomic-adds the parent refcount. The refcount bump now pins... nothing, because layers below T are already gone. The `GC_FORK_MARGIN_TXIDS` (1024 txids ≈ 1 minute, line 559) is a soft margin against a hard correctness invariant. **Recommended fix:** `fork()` and `restore_to_bookmark()` must do the refcount bump *and* the pin check in a single FDB tx with regular reads on `META/manifest.retention_pin_txid` so OCC catches the race. The atomic-add on refcount alone does not take a conflict range and cannot serialize against GC's pin advance. This is the same pattern as the lease take in the stateless spec (lines 380-388: "Regular read (NOT snapshot read)" so OCC aborts the loser).

## Major issues

7. **Bookmark "deterministic from `(timestamp_ms, txid)`" is not actually true once you include `branch_id`.** Section "Bookmarks" (line 6, line 79). Two callers calling `get_current_bookmark()` at the same `(timestamp_ms, txid)` on different branches get *different* strings (different branch_id suffix). The CF DO claim of determinism is "you can compute the bookmark from a timestamp without round-tripping" — that property requires the timestamp -> bookmark mapping to be a function. With the proposed format it isn't, because branch_id is part of the bookmark and only known after the resolve. The spec also asserts "16-byte hash-like trailing identity" matches CF DO; it doesn't — CF DO's last segment is described as "a content-addressable identity / nonce so two databases never collide" (research, line 78), one nonce per database, not per bookmark. **Recommended fix:** drop the branch_id from the bookmark wire format and instead carry it out-of-band (bookmark applies to a specific branch context, similar to how Neon's `parent_lsn` is meaningless without `parent_timeline_id`). Or commit to "bookmark is per-branch" and stop claiming CF DO equivalence.

8. **Bookmark forgery: a malicious caller can construct a bookmark for any branch_id that exists in any actor.** Per CLAUDE.md trust boundaries, the engine treats `client <-> engine` as untrusted. The bookmark wire format `{ts}-{txid}-{branch_id}` is fully attacker-constructible. `resolve_bookmark` (line 105) verifies "branch_id is reachable from this actor's branch chain" — but reachability check only fails if the branch is in a different actor. Within a single actor with multiple sibling branches (after `fork`), a caller with access to one branch can resolve any bookmark for any sibling branch and call `restore_to_bookmark` on it. The spec is silent on whether sibling-branch reads are allowed. **Recommended fix:** spell out the authorization model — either bookmarks are sibling-scoped (caller must own the branch they're targeting) or they're actor-scoped (any branch in the actor is fair game) — and add the authorization check at the engine edge per CLAUDE.md.

9. **"PITR is the same primitive as fork" breaks down because CF DO's destructive in-place restore preserves bookmark identity, and this design doesn't.** Spec line 7, 144-170. CF DO's restore replaces the DO's data; subsequent `getCurrentBookmark()` returns a bookmark *on the same DO id* with a fresh trailing hash. Application code holding old bookmarks can still resolve them (research, line 120-121). In this design, restore creates a new branch with a new branch_id; the actor's "bookmark space" suddenly bifurcates. The undo bookmark (line 159, 168) the spec claims survives — does survive — but only because the spec retains the old branch as a sibling. That works for one undo. **What about an undo of an undo of an undo?** Each restore creates another branch; the actor accumulates sibling branches forever, paying retention cost for all of them. CF DO has no such accumulation because the in-place restore overwrites. **Recommended fix:** explicitly document the divergence from CF DO and the cost model (every undo creates a billable sibling), and consider exposing a `restore_to_bookmark_destructive` (mentioned in future work line 662) as the actual CF-DO-equivalent path with the old branch deleted on confirmation.

10. **Cold compactor's "disjoint META sub-keys" claim ignores `META/quota`.** Lines 540-549. Cold compactor decrements `META/quota` (line 524: "Cold compactor decrements when DELTAs are drained"). Hot compactor also writes `META/quota` (`atomic_add ... -bytes` per the stateless spec line 213). The spec admits both compactors do quota mutation but doesn't enumerate quota in the disjoint-sub-key list. The fact that atomic-add doesn't take a conflict range means correctness is preserved, but the "disjoint META sub-keys" framing is misleading. **Recommended fix:** state explicitly that quota mutations are atomic-add and therefore commute, not that they're tx-disjoint.

11. **Storage-only-scope leaves swap-pointer atomicity undefined.** Section "Storage library scope only" (line 11) and "Point-in-time recovery" (lines 144-170). `restore_to_bookmark` returns a new branch_id; pegboard is supposed to swap the actor's head pointer. **What if pegboard crashes between the branch creation and the head swap?** The spec says "the original branch is **not deleted by restore**" (line 170), so the data isn't lost. But the user observed "I called restoreToBookmark and the actor came back up unrestored" — confusing UX. The undo bookmark only saves you if you know *which* branch is now live. **Recommended fix:** either (a) the storage layer exposes a single atomic `swap_head_to_bookmark` op that does both fork-create and head-swap in one FDB tx (cheap, since both are metadata-only), or (b) explicitly document that broken half-swap is the broader system's problem and provide a `recover_swap` API. Currently the spec offloads atomicity to a non-existent layer and ignores the failure mode.

12. **Layer file `min-max-checksum.ltx` filename collisions on overlapping ranges.** Section "Layer files" (line 285, 298). LiteFS's `<minTXID>-<maxTXID>.ltx` works because LiteFS has linear txid order on a single timeline. With branches, the same `(min_txid, max_txid)` range can be produced by two different branches' compactor passes (they each have independent txid sequences? — the spec doesn't say if txid is per-branch or globally monotonic). And L1/L2/L3 compaction will produce overlapping ranges by design (line 442-443: "Merge contiguous L0 layers within a 30-second wall-clock window"). The checksum suffix probably saves you, but the spec doesn't say so explicitly. **Recommended fix:** clarify that txid is per-branch (which makes the layer naming branch-scoped via the S3 prefix `branches/{branch_id}/`), and explicitly note the checksum disambiguates same-range outputs from different compaction levels. Also: nothing prevents two parallel cold compactor passes from emitting the same `(min, max)` range with different content if leases are misacquired. Add an assertion that the cold lease must be held before any layer file write.

13. **Metering: "v1: bill total cold_bytes per branch" double-charges grandchild forks.** Future work (line 661): "A forked branch shares unmodified pages with its parent; a naive byte count would double-bill ... v1: bill total cold_bytes per branch; revisit when pricing matters." With grandchildren, you triple-charge the bytes that all three share. With 100 forks of one DB, you charge 100x. This isn't "fix in v2" territory — it directly affects the customer's bill. CF DO has the same problem in a sense (each DO has its own cold tier), but CF DO doesn't expose forking. Neon explicitly bills only the delta from parent (research, line 96: "Storage billed only for delta between parent and child"). **Recommended fix:** at minimum, document the bill-shape clearly and gate fork-creation behind a "you understand storage costs" toggle. Better: bill cold bytes by the layer files actually owned by this branch (excluding pinned-but-not-owned parent layers), so a fresh fork starts at 0 cold bytes and only accrues as it diverges. This matches Neon and is implementable today without v2.

## Minor issues

14. **`get_bookmark_for_time(t)` algorithm is underspecified for cross-branch resolution.** Line 99-100: "Walks the parent chain if t predates this branch." But the bookmark itself encodes `branch_id`, so walking the parent chain produces a bookmark on *the parent* branch — not on `self`. Is that the intent? If so, the returned bookmark is one the caller can't directly use to fork from `self` — they'd have to fork from the parent. Document this.

15. **`SHARD/{shard_id_be:4}` keying with a 4-byte shard_id (line 236) limits one branch to 2^32 shards.** Stateless spec uses `SHARD_SIZE = 64` pages and `PAGE_SIZE = 4096` bytes (≈256 KB per shard). 2^32 shards × 256 KB = 1 PB. Fine, just note it. But branch's hot-tier cap is 10 GiB (line 524), so 2^32 is wildly over-provisioned. Consider `:2` or `:3` shard_id encoding to save key bytes per row, especially since `PIDX/delta/{pgno_be:4}` is also 4 bytes.

16. **`COMMITS/{txid_be}` retention is split between hot and cold but the hand-off invariant is hand-waved.** Lines 432-433: cold compactor "delete `COMMITS/{T}` for T < retention_pin_txid (kept for txid->wall_clock map; cold tier has the same data via `BookmarkIndex`)." But the `BookmarkIndex` is sparse (line 341: "every N commits or every M seconds"). Resolving an exact txid -> wall_clock for a deleted hot COMMITS entry requires interpolation between two cold entries, which is approximate, not exact. A bookmark's `checksum` field requires the exact CommitRecord. **Recommended fix:** either keep `COMMITS` exact in cold (1:1 dump alongside layers), or document that bookmarks resolved from cold-only have approximate timestamps and exact-txid checksums require a layer-file scan.

17. **Refcount race on simultaneous `delete_branch` + `fork`.** Line 562-564 ("Branch delete during read") only addresses reads. What about: child branch C exists, refcount=1. Concurrently: caller A calls `delete_branch(C)`; caller B calls `fork(at child of C)`. A's tx decrements refcount to 0 and clears C's hot tier; B's tx adds a new grandchild and bumps... what? B reads C's manifest, sees Live, writes new branch with `parent=C`. C is gone. The new grandchild is orphaned. **Recommended fix:** `fork` reads parent's `state` field with a regular read so OCC catches the concurrent delete. Or `delete_branch` first transitions to `Deleted` state, and `fork` rejects parents in `Deleted` state.

18. **`CF DO claim accuracy`: divergences from CF DO, ranked.** The spec says "closest to CF DO" but doesn't enumerate divergences. Ranked by how confused a CF DO user would be:
   - **(1) Forking is exposed.** CF DO has no fork API. A user expecting CF parity will not look for `fork`. (Acceptable, but document.)
   - **(2) Bookmark wire format differs.** CF: `0000007b-0000b26e-00001538-<hash>` (4 segments). This: `{ts}-{txid}-{branch_id}` (3 segments). Lex order matches across branches per spec but actually doesn't (see issue 4). Different segment count is enough to break any tooling that tries to parse the bookmark.
   - **(3) Restore is not destructive in-place.** CF DO's `onNextSessionRestoreBookmark` overwrites the DO's data. This spec creates a sibling branch and pegboard swaps the pointer. Bookmarks taken before the restore continue to resolve to the old (now-sibling) branch's data. Different observable behavior on `getCurrentBookmark()` after restore: CF returns a fresh bookmark on the same DO; this returns a bookmark on a different branch_id. (See issue 9.)
   - **(4) `getBookmarkForTime` is not a function of (timestamp) alone.** Per issue 7, branch context is required.
   - **(5) Restore granularity.** CF DO is per-commit (research line 114). This is per-commit (line 17, line 21). Match.
   - **(6) Retention default.** CF DO 30d (paid) / 7d (free). This 30d default. Match for paid.

## Direct contradictions found

- **Line 7 ("PITR is the same primitive as fork")** vs **line 7 swap-clause + line 170 ("the original branch is not deleted by restore")**: if PITR is fork-with-swap, the old data is still there as a sibling. CF DO destructive restore (research line 119) deletes the old data on next boot. The spec calls itself CF-DO-style and Neon-style simultaneously; these are different primitives.
- **Line 11 ("Storage library scope only")** vs **line 162-167 (pseudocode does `pegboard::swap_actor_branch` and `ctx.abort()`)**: the swap is described inline as if it's part of the storage flow, but is asserted out of scope. Either the swap is part of the contract or it isn't.
- **Line 411 ("They never read/write the same DELTA")** vs **lines 542-543 (hot writes DELTA/*; cold reads-then-clears DELTA/*)**: trivially false as stated. The intended invariant is "they don't write the same DELTA in tx-overlapping windows" — but as worded the claim is wrong. (See issue 5.)
- **Line 458 ("never advance the pin past what was already set")** vs the implication of the surrounding `min(pin, C.parent_txid)` arithmetic: monotonic pin makes child-delete unable to release space (issue 3).
- **Line 6 / open-q line 647** "lex ordering across branches" — spec asserts it works, open question admits it's only "fine" if you accept "by wall clock then by branch identity" as the order. These can't both be true; the open question is the actual behavior, and the goal claim should be retracted.

## Stuff that's actually right

- **Treating fork as metadata-only with refcount-based pin extension** is the correct primitive. It directly maps to Neon's `ancestor_timeline_id`/`ancestor_lsn` and is the right shape for "fork is free, divergent storage costs." (Lines 119-142.)
- **Splitting hot and cold compactor into separate leases on disjoint META sub-keys** is the right concurrency model and matches the stateless spec's pattern. The hot-compactor lease design is good prior art to copy. (Lines 537-549, modulo issue 5.)
- **CRC-ISO-64 rolling checksum + storing pre/post checksum on layer files** is the correct verification primitive borrowed from LiteFS/Litestream and is the right call given that branch parent_checksum is the reachability invariant.
- **The decision to store `COMMITS/{txid}` in hot for fast `get_bookmark_for_time` and a sparse `BookmarkIndex` in cold for older lookups** is a reasonable hot/cold split; the bookmark API ergonomics depend on this.

## Top-line recommendation

The branch primitive and overall architecture are sound but the spec underspecifies three correctness-critical interactions: (a) deep-chain fall-through cost and depth bound, (b) GC pin computation in the presence of frozen-sibling branches and child-delete reclamation, and (c) the atomicity boundary between fork-create and pointer-swap. These are not "fix in v2" issues; they will silently corrupt bills, retention, or cause customer-visible data loss after a few real fork operations. Fix the GC predicate, document the CF DO divergences honestly, and either bring the swap into the storage atomic boundary or specify the recovery API.
