# Architecture review: sqlite-rough-pitr.md

Reviewer mandate: find architectural holes. Specific, citation-bearing findings only.

## Critical issues

1. **Namespace fork "lazy parent-chain resolution" cannot work as specified — actor pointers are global, not per-namespace.** §4 (architectural overview) and §8.1 say a fork_namespace lazily snapshots the actor catalog by falling through `NSBRANCH/.../actors/{actor_id}` from N' to N. But `APTR/{actor_id}/cur` is a *single global key per actor_id* (§6, lines 222-223). After fork_namespace creates N', when actor A in N gets a write, the live `APTR/A/cur` pointer mutates. Any read in N' that "falls through to parent" then resolves through the same APTR and sees the post-fork state of A. There is no captured snapshot anywhere. The spec waves at this as "lazy parent-chain resolution" but never says how N' pins the per-actor pointer values it inherited from N.

   Reproduction: create namespace N with actors A (branch B_A) and B (branch B_B). At versionstamp V, fork_namespace(N) → N'. After V, in N: rollback_actor(A) flips APTR/A/cur to a new branch B_A2. A read on actor A in namespace N' now resolves APTR/A/cur → B_A2, which is the post-V state, defeating the namespace fork.

   Fix: either eagerly write per-(NS-branch, actor_id) pointer records in `NSBRANCH/.../actors/{actor_id}` at fork time (giving up O(1) fork on a >1-actor namespace), or change the model so APTR is keyed by `(namespace_branch, actor_id)` rather than global `actor_id`, or store the captured `(actor_id → ActorBranchId, parent_versionstamp_cap)` snapshot lazily on first observation. The spec needs to pick one; right now it asserts impossible semantics.

2. **`ensure_tier_at_least(tx, source.namespace_branch, ...)` is performed against an immutable record, but tier transition writes to that same record.** §8 (lines 401, 464) and §14 (lines 902-907) say the first fork on a Tier-0 namespace atomically promotes to T1 in the same tx. Yet `Tier` is a field on the immutable `NamespaceBranchRecord` (§5.3, line 151) — the spec explicitly states branch records are immutable except for deletion (§5.5 line 195). You cannot atomically flip an immutable record's field. Either tier must move out of the immutable record into a mutable counter-style key (parallel to `refcount`/`desc_pin`), or "immutable" is a lie. Pick one.

3. **GC pin formula uses a non-existent field.** §13 line 884 reads `refcount > 0 ? branch.created_at_versionstamp : SENTINEL_INFINITY`. There is no `created_at_versionstamp` field on `ActorBranchRecord` (§5.4) or `NamespaceBranchRecord` (§5.3). Both records carry `created_at_ms` (wall clock) and `parent_versionstamp` (the fork-point in *parent*, not self). The branch's own root versionstamp is never recorded. As a result, "anything older than my own root" is not computable from data on the record. Either add a `root_versionstamp` field, or change the formula to use `parent_versionstamp` (which only works for non-root branches), or admit the formula is `min(desc_pin, bk_pin)` with `refcount` only governing whether the branch itself is deletable.

4. **Versioned SHARD read race against eviction is undefined.** §5.2 (line 124) says reads do `reverse-range scan SHARD/{shard_id}/ constrained by as_of_txid <= V, return first hit`. §12.3 (line 864) says eviction does `clear_range` on evictable shard versions. There is no synchronization between an in-flight `get_pages` snapshot read and a concurrent eviction `clear_range` — the eviction tx commits, the live reader's snapshot sees nothing at the just-cleared key, and the read silently falls through to the *next-older* shard version (which has stale data for those pages). The spec needs to either (a) require ancestor pinning during in-progress reads (similar to mvSQLite's pin-token), (b) gate eviction on a "no live read tx older than X" predicate, or (c) require readers to validate the chosen `as_of_txid` against `META/head.head_txid` and retry if it appears too old. Currently nothing prevents a torn read.

5. **`derive_branch_at` returns a branch with no `/META/head` and no clear way to read at `at_versionstamp`.** §8 line 437-438 says "the new branch has no /META/head until its first write" and "Reads on the new branch fall through to source via parent_versionstamp." But the read path (§10) reads `/META/head` of the current branch *first* to get `head_txid` and `db_size_pages` (line 730). On a freshly forked branch this key is absent. The spec doesn't say what to do — fall through to parent's `/META/head` would expose post-fork commits of the source. It must be either (a) seed a `MetaHead` snapshot of the parent's head-as-of-versionstamp at fork time (a write — fork is no longer purely metadata-O(1) for the *commit*), or (b) define explicit "synthetic head" semantics where `head_txid = parent_head_at_versionstamp` and `db_size_pages` is captured at fork. Without one of these, a fresh fork cannot answer `db_size_pages`.

6. **Atomic-min `desc_pin` is set to `at_versionstamp` instead of `parent_versionstamp`-bound.** §8 line 434 (`derive_branch_at`) and §8.1 line 484 do `atomic_min(BRANCHES_list_desc_pin_key(source_branch), at_versionstamp)`. This pins the *source* branch's GC at `at_versionstamp` so the source can't GC history below the fork point. But the inherited prior-art spec (§ "Forking" of `sqlite-pitr-fork.md` line 174) does `atomic_min(parent.oldest_descendant_parent_txid, point.txid)`. The new spec replaces txid with versionstamp without explaining how that 16-byte versionstamp value compares against COMMITS' `txid_be:8` keys for GC purposes. GC pass needs to map "delete COMMITS row at txid T iff `versionstamp(T) < gc_pin`" but the spec never defines the inverse mapping (versionstamp → txid) for GC scans. Either store both txid and versionstamp on each pin-bearing key, or use txid throughout (and accept that cross-branch ordering is weaker), or define the index that makes versionstamp → COMMITS lookup tractable.

7. **Tier 0 → Tier 1 transition raced against a concurrent commit cannot be made atomic without writing on the commit hot path.** §14 line 907 ("set_if_equal(tier, T0, T1)") describes transitioning the *namespace branch's* tier when a fork lands. But the actor branch record (§5.4 line 179) also carries `tier`. When a Tier-0 namespace promotes to Tier-1 at moment M, every Tier-0 actor branch in the namespace still carries `tier: T0`. The spec says tier is "inherited from parent on fork" (§5.3 line 145) but doesn't say what happens to *existing* records. Hot compactor's tier-aware DELTA deletion (§12.1) reads the actor branch's tier; if it reads T0, it unconditionally deletes folded DELTAs even though the namespace is now T1 and cold compactor expects them. Either tier must be read off the *namespace* branch on every hot-pass decision (extra read), or all existing actor branch records must be rewritten on T0→T1 promotion (O(N) work on what is supposed to be O(1)). Spec must commit.

## Major issues

8. **Eviction predicate ignores indirect descendants across namespace forks.** §12.3 lines 858-863 list the predicate gates. Charter scenario 5 plays out: namespace N has actor A (branch B_A); fork_namespace(N) → N' at versionstamp V. By design, A is *not* eagerly registered in N'. So `BRANCHES/list/B_A/desc_pin` never gets the atomic-min from the namespace fork — only from `fork_actor`. Now in N, A is left untouched after V. Eviction sweep on B_A reads `desc_pin = SENTINEL_INFINITY` (no fork_actor pin recorded). It evicts SHARDs `as_of_txid ≤ V`. N' later boots actor A and reads APTR/A/cur → B_A → ancestor read at `parent_versionstamp` cap = V. The page data behind those pre-V SHARDs is gone. Fix: namespace fork must atomic-min `desc_pin` on every actor in the source namespace's catalog (impossible to do lazily — the namespace doesn't know which actors exist in advance). This is the same root cause as Critical #1.

9. **Bookmark resolution algorithm doesn't bridge namespace forks.** §9 lines 700-713 walks the *actor* parent chain. But after `fork_namespace(N) → N'`, an actor A's "current" branch in N' is still inherited via lazy resolution from N. A bookmark created on actor A in N' before the next mutation has no actor-branch parent record describing the namespace edge (because no actor branch was forked). Walking actor parents from B_A upward lands you in N's history, not N's-as-of-V. The resolver returns commits that happened in N *after* V, which the caller in N' should not be able to observe. Fix: resolution must additionally cap by the namespace branch's `parent_versionstamp` chain. Spec doesn't model this.

10. **`COMPARE_AND_CLEAR` invariant carry-over is not extended to versioned SHARDs.** Inherited stateless spec uses `COMPARE_AND_CLEAR` for PIDX deletes (CLAUDE.md line 32). New spec adds versioned SHARD deletes via `clear_range` on eviction (§12.3 line 864) and DELTA deletes (line 865). Eviction can race a hot compactor pass that just produced a new SHARD version `T+1`: eviction's regular-read of `desc_pin`/`bk_pin` happens at time T, hot pass commits `T+1` at time T+ε, eviction `clear_range`'s the *folded* DELTA range that the hot pass just sealed — losing pages T+1's SHARD doesn't yet contain. Spec needs an OCC fence on `BranchManifest.last_hot_pass_txid` analogous to the `cold_drained_txid` fence in §12.2. Currently eviction has only the global lease, which serializes evictions vs each other but not vs hot.

11. **`derive_branch_at` does not capture `db_size_pages` at the fork point — it captures live state.** The fork primitive does not record what `head.db_size_pages` was at `at_versionstamp`. Reads on a fresh fork must compute db_size_pages somehow, but the source's `META/head` is mutable and reflects post-fork state. The spec implicitly relies on the parent fall-through reading source's *current* head, which gives size as of "now" not as of V. Caller in the new branch will see pages that didn't exist at V (because source grew the DB after V). Either store `(head_txid_at_v, db_size_pages_at_v)` snapshot on the new branch record, or require fork callers to scan COMMITS up to V to find the `db_size_pages` value at V (an extra O(commits) read). Spec is silent.

12. **"Storage-only-scope" leaves rollback-vs-cache-invalidation window with undefined semantics.** §1 goal 7 ("Storage layer exposes primitives only. Cache invalidation on rollback... are upstream concerns"). After the rollback tx commits in FDB, ActorDb instances that already cached the old `current_branch` will keep serving reads/writes against the *old* branch — including writing new commits to the old branch's `/META/head`. Pegboard exclusivity does not save you here, because the WS conn that holds the cache is the legitimate exclusive owner and has no signal that its pointer became stale. The spec's only nod to this is §10 line 751 ("cache invalidated when APTR changes, detected by including a fast APTR.last_swapped_at_ms check in the same tx as /META/head"). But this read happens on every `get_pages`/`commit`, so the assertion that branching adds nothing to the hot path is wrong: every op now reads APTR. And worse, between the rollback commit and the reader's next op, the in-flight commit may already be written to the old branch — undoable only by manual cleanup. Spec needs to either (a) accept that rollback requires upstream actor restart before it's usable (state this clearly), or (b) define a "rolled-back commits go to limbo, not the new branch's head" rule, or (c) add the APTR read to the hot path and cost it accordingly.

13. **Versionstamp → COMMITS-txid binary search is hand-waved.** §11 line 774 says commits write `SetVersionstampedValue` against `COMMITS/{txid_be:8}`. §9 (resolution) and §13 (GC) need the inverse: given a versionstamp V, find the largest `txid` whose recorded versionstamp is `≤ V`. There's no index from versionstamp to txid. Resolution would need a full COMMITS scan per resolution. The prior fork spec used txid throughout for exactly this reason. Either add `VTX/{versionstamp_be:16}/{txid_be:8} → empty` as a secondary index (extra hot-path write per commit), or admit resolution is O(branch commits) scan. Spec doesn't admit either.

14. **Pinned bookmark S3 PUT is "synchronous in the issuing pod" but the issuing pod is `pegboard-envoy`, and pegboard-envoy is documented as the trust boundary.** §17 line 989-991: "The full-DB image LTX encoding happens in the issuing pod; the PUT travels at S3 speeds. For a 1 GiB DB at typical S3 PUT speeds, wall-clock is seconds to tens of seconds. Caller blocks." Pegboard-envoy is the WS handler. A multi-second blocking S3 PUT on the WS conn handler stalls every other actor multiplexed onto that conn (the WS conn manages per-actor `ActorDb` via `scc::HashMap` per CLAUDE.md line 22). Spec does not say `create_pinned_bookmark` runs on a separate task or pod. Either the cold compactor pod handles it (out of band), or the WS conn must spawn it on a worker pool. Either way, the contract "caller blocks" needs a transport story — currently the storage layer just returns, but the path to S3 is unspecified.

15. **Layer-file naming with `as_of_txid` packed into image filename leaves multi-shard packing undefined.** §7 lines 324-325 specifies `image/{as_of_txid_high_bytes_hex:8}/{shard_id_be_hex:8}-{as_of_txid_be_hex:16}.ltx`. Each file holds one shard at one as_of_txid — spec implies one-shard-per-file but never states it explicitly. CLAUDE.md inherited bullet (line 48) says "Layer filenames omit content checksum (`delta/{min_txid}-{max_txid}.ltx`). Re-uploads after lease loss overwrite cleanly." This idempotency guarantee depends on the bytes being deterministic for a given filename — if a packing strategy is ever introduced, two passes with different shard-set choices could produce different bytes for the same filename. Spec should explicitly add the constraint "exactly one shard per image file; multi-shard packing is forbidden."

## Minor issues

16. **`ResolvedVersionstamp` is referenced but never defined.** §8.1 line 453, §8.2 line 510, §8.3 line 558, §8.4 line 608 all use this type as a parameter. The struct is never declared. From context it appears to be `{ versionstamp: [u8; 16], bookmark: Option<BookmarkRef> }` — declare it in §5.

17. **`APTR/.../history/{ts_ms_be:8}` collides on sub-millisecond rollbacks.** §6 line 223. Two rollbacks within the same millisecond would write to the same key, silently overwriting the audit log entry. Use `{ts_ms_be:8}{nonce:4}` or write versionstamp to make uniqueness intrinsic.

18. **`desc_pin`/`bk_pin` initial value semantics undefined.** Atomic-min on an absent key in FDB has driver-specific behavior. Spec should say "absent treated as `0xFF...FF` (sentinel infinity)" or similar, otherwise the first atomic-min produces driver-dependent results. Same applies to `oldest_descendant_pin`.

19. **`schema_version: u32` on `NamespacePointer`/`ActorPointer`/`BranchCounters` adds wire bytes for no benefit.** These records mutate constantly. CLAUDE.md (inherited, line 35) requires schema_version on *persisted S3 objects*. Pointers are FDB-resident, not S3 persisted. Drop the field on internal-only records or be consistent and add it to `DBHead` and `BranchManifest` too (which already have it).

20. **`swapped_once: bool` on `ActorPointer` is dead code.** §5.4 line 188 says it distinguishes "never swapped" from "swapped to the same branch". But (a) you can't swap to the same branch (rollback always allocates a new branch — §8.4 line 610), (b) `last_swapped_at_ms` is initialized at create time to `now_ms()` so it's never zero. Field is unused; remove or document a use case.

21. **MAX_NAMESPACE_DEPTH is not enforced anywhere.** §24 OQ 1 lists this as open, but namespace fork (§8.1) has no depth check, only the actor primitive does. Without a cap, namespace fork chains can grow unboundedly and namespace ancestry walks (needed for catalog resolution per Critical #1) become O(depth). Add a constant and enforce.

22. **"Pegboard wired to be notified of pointer flips (out-of-band)" is undefined.** §22 line 1092. No mechanism specified — UPS subject? FDB watch? Direct call from rollback into pegboard? This is the load-bearing piece of "swap is upstream's concern" and it's punted entirely.

23. **`BranchManifest.last_access_ts_ms` is mutated lazily by `ActorDb` on read/write activity (§12.3 line 850).** The stateless spec guarantees `get_pages` does zero writes (§ "Hot-path key reads" table). Adding a lazy `last_access_ts_ms` write contradicts that; even "best-effort, not load-bearing for correctness" still costs an FDB write per access. Spec needs to specify a "throttled best-effort" with a write-skip predicate (e.g., "skip if `now - cached_value < ACCESS_TOUCH_THROTTLE_MS`"), or move this to a separate non-tx write path.

## Direct contradictions found

- **Tier on immutable record vs atomic transition.** §5.3 line 151 (`tier: Tier` on `NamespaceBranchRecord`) + §5.5 line 195 ("Branch records are immutable except for deletion") vs §14 line 907 ("atomically promotes to T1 in the same tx"). Cannot all three be true.

- **Hot path is stateless / no extra reads vs APTR check on every op.** §1 goal 6 ("PIDX flattened-ancestry cache is the only addition") + §10 line 751 ("APTR.last_swapped_at_ms check in the same tx as /META/head"). The APTR read *is* an addition; goal 6 understates the hot-path cost.

- **`BranchManifest` `cold_drained_txid` location.** §6 line 244 says it lives on `META/manifest`. §6 line 240 also assigns `cold_drained_txid` to `META/cold_compact`. §12.2 Phase C line 819-820 writes both. The spec is duplicating the value across two keys. Either pick one source-of-truth or document why both copies exist (and how they're kept consistent).

- **"Forks contribute +1 from the child branch's parent reference"** (§5.5 line 202) vs `derive_branch_at` only doing `atomic_add(refcount, 1)` on source (§8 line 429). The doc-comment says +1 from the *child*; the code says +1 on the *source*. Same effect but opposite framings; reconcile the wording.

- **Cold compactor tier-aware DELTA deletion.** §12.1 line 785 says hot compactor leaves DELTAs alone in Tier 1+. §12.2 phase C never explicitly deletes DELTAs from FDB after upload. Spec line 845 says eviction handles "DELTA covered by an uploaded delta layer", but the contract for "who actually frees folded-and-uploaded DELTAs" is split across two compactors with no precise handoff.

## Parity check vs inherited specs

- **Stateless spec / CLAUDE.md says hot path is stateless and APTR is not mentioned.** This spec adds APTR and an APTR check on every op. The justification ("rollback safety") is reasonable but the divergence is unacknowledged. Update CLAUDE.md or call it out in §3 ("inherited constraints") explicitly.

- **Stateless spec: "`/META/head` exists at the start of its tx... seed if absent" (lines 260-261).** New spec preserves this for first commit on root branch but adds *fork* as a second source of "META/head absent" state. The first-commit init logic must now also distinguish "first commit ever, parent has no data" from "first commit on a fork, parent has data we should fall through to". Spec doesn't update the init logic to handle this.

- **CLAUDE.md inherited bullet: "`COMMITS/{txid_be}` records `wall_clock_ms` only".** This spec extends `CommitRow` to `{wall_clock_ms, versionstamp}` (§6 line 273-274). Strict divergence from the inherited contract. Update CLAUDE.md to match or acknowledge. (The prior PITR spec, in fact, *removed* the checksum from COMMITS to keep wall_clock_ms only — this spec adds versionstamp back.)

- **CLAUDE.md inherited bullet: "Frozen branches use `created_at_ms`-based retention".** New spec eliminates the `BranchState::Frozen` enum entirely (no mention in §5). The retention semantics for "an actor that was rolled back, source still pointed at by a bookmark" must replicate Frozen behavior somehow. Spec is silent. Either define the new equivalent or restore Frozen.

- **CLAUDE.md inherited bullet: "PIDX deletes use FDB `COMPARE_AND_CLEAR`".** New spec doesn't mention `COMPARE_AND_CLEAR` for the new versioned-SHARD deletion path. By analogy it should apply, but the spec needs to extend the bullet to "PIDX deletes AND eviction-of-versioned-SHARD deletes use COMPARE_AND_CLEAR" or argue why eviction is OK without it.

## Stuff that's actually right

- **Single derive-from-source primitive shared by all four operations** (§8) is a clean factoring. It does cut the surface area meaningfully.

- **OCC fence on `bk_pin` (regular-read inside fork tx) instead of time margin** (§8 line 409, §15 line 936) is exactly right and matches what the prior PITR review demanded. It's a real correctness improvement.

- **Tier 0 with zero S3 cost** is a strong product story; the Tier 0/1/2 ladder maps cleanly onto the cold compactor and eviction compactor being conditionally enabled. The cost ladder is honest.

---

Top concerns summary (3 bullets, <100 words):

- Namespace fork's "lazy parent-chain resolution" is incoherent because actor pointers are global; rollbacks/forks in N after fork_namespace bleed into N'. This breaks the central two-level indirection promise.
- Tier field is on the immutable branch record but the spec demands atomic T0→T1 promotion in a single tx, and propagation to existing actor branch records on namespace promotion is unspecified.
- Versioned SHARD model lacks an OCC fence between eviction and concurrent hot/in-flight reads, so torn reads against a just-cleared-and-replaced shard version are possible. Same gap exists for namespace-fork-induced descendant pins on actors that were never explicitly forked.
