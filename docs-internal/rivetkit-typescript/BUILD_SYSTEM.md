# TypeScript Build System

## Overview

Each TypeScript package in the monorepo uses a two-stage build:

1. **tsup** bundles JavaScript (ESM + CJS) with `dts: false`
2. **tsc** generates `.d.ts` declarations separately via `tsconfig.build.json`

The `sync-package-json.ts` script auto-generates exports, build scripts, tsconfig.build.json, and standard fields from a single `"rivetkit"` config block in each package's `package.json`.

## Why Separate DTS Generation

tsup uses rollup-plugin-dts internally, which generates hashed chunk `.d.ts` files (e.g., `config-Kp8zDf33.d.ts`) when `splitting: true`. These hashed filenames:

- Change on every build
- Have no public import path in the package exports
- Break downstream projects using `composite: true` or `isolatedModules: true` with TypeScript's "inferred type cannot be named" error

Generating declarations with `tsc --emitDeclarationOnly` produces stable, per-file `.d.ts` files that mirror the source tree structure. `tsc-alias` rewrites `@/*` path aliases in the emitted declarations (only needed for the `rivetkit` package).

## The `rivetkit` Config Block

Each managed package has a `"rivetkit"` block in its `package.json`:

```json
{
  "rivetkit": {
    ".": "src/mod.ts",
    "./client": "src/client/mod.ts"
  }
}
```

- Keys starting with `.` are entry points (mapped to exports)
- Other keys are config options

### Config Options

| Option | Default | Description |
|--------|---------|-------------|
| `outDir` | `"dist"` | JS output directory (tsup) |
| `typesDir` | `"dist/types"` | Types output directory (tsc) |
| `rootDir` | `"src"` | tsconfig.build.json rootDir. Use `"."` for packages importing from outside `src/` |
| `preBuild` | — | Command to run before tsup (e.g., `"pnpm run compile:bare"`) |
| `postBuild` | — | Command to run after tsc |
| `browserOutDir` | `"dist/browser"` | Browser build output. Set equal to `outDir` to include browser entries in main tsup build |
| `extraIncludes` | `[]` | Additional include globs for tsconfig.build.json |
| `extraExcludes` | `[]` | Additional exclude globs for tsconfig.build.json |

### Entry Types

- **String**: `".": "src/mod.ts"` - standard entry
- **Browser**: `".": { "default": "src/mod.ts", "browser": "src/mod.browser.ts" }` - adds browser condition to exports
- **Static**: `"./tsconfig": { "static": "dist/tsconfig.json" }` - passthrough, no bundling

### What the Sync Script Generates

For each package with a `rivetkit` block:

- **`exports`** in package.json - maps entry points to JS and `.d.ts` output paths
- **`scripts.build`** - the full build command
- **`tsconfig.build.json`** - tsc config for declaration emission with `composite: true` and project references
- **Standard fields** - `type`, `engines`, `sideEffects`, `files`

## Build Pipeline (Per Package)

```
[preBuild] && tsup [entries] && tsc -p tsconfig.build.json --noCheck [&& tsc-alias]
```

1. **preBuild** - schema compilation, code generation, etc.
2. **tsup** - bundles JS (ESM + CJS) with `dts: false`, `splitting: true`
3. **tsc** - generates `.d.ts` files with `--noCheck` (skips type checking, just emits declarations)
4. **tsc-alias** - rewrites `@/*` path aliases in `.d.ts` files (only for packages that use `@/` imports)

`--noCheck` is used because type checking is done separately via `check-types`. This keeps the build fast and avoids issues with test fixtures.

## TypeScript Configuration

### Three tsconfig levels:

- **`tsconfig.base.json`** (repo root) - shared base config
  - `strict: true`, `isolatedModules: true`, `skipLibCheck: true`
  - `moduleResolution: "bundler"`, `target: "esnext"`
  - All packages extend this (except excluded packages)

- **`tsconfig.json`** (per-package) - development config
  - Used by IDE for type-checking, includes tests/fixtures
  - May have `@/*` path aliases and cross-package path mappings
  - `noEmit: true` for frontend packages

- **`tsconfig.build.json`** (per-package, generated by sync script) - build config
  - `composite: true` - enables incremental compilation and project references
  - `emitDeclarationOnly: true` - only generates `.d.ts` files
  - `declarationMap: true` - enables "go to definition" in consumers
  - `skipLibCheck: true` - skips checking `.d.ts` from dependencies
  - Tight `include`/`exclude` - only `src/**/*`, excludes tests
  - `references` - auto-generated from workspace dependencies

### Why `composite` is on `tsconfig.build.json` only

Development tsconfigs include tests, fixtures, and scripts that don't compile cleanly with `composite`'s strict requirements. The build tsconfig has tight boundaries (`src/**/*` only) making it compatible with `composite`.

### `isolatedModules`

Ensures each file can be transpiled independently (required by tsup/esbuild). Catches:
- `const enum` across file boundaries
- Re-exports of types without `type` keyword
- Namespace merging across files

### Project References

tsconfig.build.json files reference their workspace dependencies' tsconfig.build.json files. This enables:
- Incremental `tsc --build` (only re-checks changed packages)
- Native dependency graph resolution by tsc

References are auto-generated by the sync script from `dependencies` and `devDependencies` in package.json.

## Build Orchestration

Turbo handles build orchestration via `turbo.json`:
- `build` depends on `^build` (all workspace dependencies build first)
- `check-types` depends on `^build` (types from dependencies must be available)
- `test` depends on `^build` and `check-types`

## Dev Workflow

- **IDE**: uses `tsconfig.json` (development config). Reads source directly across packages via path aliases. No build step needed for intellisense.
- **Rebuild a package**: `pnpm turbo build --filter=<pkg>`
- **Type check**: `pnpm turbo check-types --filter=<pkg>`
- **Incremental DTS**: `tsc --build` from a package directory (uses `composite` + references)

## Adding a New Entry Point

1. Add the entry to the `rivetkit` block in package.json
2. Run `tsx scripts/sync-package-json.ts`
3. Verify with `tsx scripts/sync-package-json.ts --check`

## Adding a New Package

1. Create the package directory under one of the `PACKAGE_DIRS`
2. Add a `rivetkit` config block to its package.json
3. Run `tsx scripts/sync-package-json.ts` to generate exports, build script, and tsconfig.build.json
4. Add `tsup.config.ts` that imports from `tsup.base.ts`

## Lefthook Integration

The sync script runs as a pre-commit check:
```yaml
sync-package-json:
  glob: "**/package.json"
  run: tsx scripts/sync-package-json.ts --check
```

## Excluded Packages

These packages are NOT managed by the sync script:

- **`@rivetkit/engine-api-full`** - generated API, standalone legacy tsconfig with `moduleResolution: "node"`
- **`@rivetkit/sqlite-vfs-test`** - private test package, no build/exports
- **`@rivetkit/mcp-hub`** - private, fully custom tsup config, depends on non-rivetkit workspace package
- **`@rivetkit/devtools`** - private IIFE bundle, `dts: false`, no type consumers
