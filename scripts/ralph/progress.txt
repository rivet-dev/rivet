## Codebase Patterns
- SqliteRuntimeDatabase interface only has exec/run/get/all/close methods. No prepare() or columns(). Use all() which returns Record<string, unknown>[] then Object.keys() to get column names.
- KV databases use #actorKvDatabases map with lazy open via #getOrCreateActorKvDatabase. App databases follow the same pattern with #actorAppDatabases.
- Actor teardown happens in sleepActor (close db, delete runtime, remove entry) and destroyActor (close db, delete files, reset entry). Both paths must handle app databases.
- Bridge keys in DYNAMIC_HOST_BRIDGE_GLOBAL_KEYS are string constants prefixed with __rivetkitDynamicHost.
- Pre-existing typecheck errors exist in fixtures/workflow.ts, engine driver, and test files. These are not related to dynamic actor changes.
- There is no host-runtime.ts. Both isolated-vm and secure-exec use the same #setIsolateBridge() in isolate-runtime.ts.
- New ActorDriver methods should be optional (?) and delegate to FileSystemGlobalState in FileSystemActorDriver.
- The isolate-side actorDriver object lives in dynamic-isolate-runtime/src/index.cts (not src/dynamic/host-runtime.ts). Its DynamicActorDriver is a locally-typed subset of ActorDriver.
- drizzle sqlite-proxy expects positional arrays from the callback; RawDatabaseClient returns objects. Use Object.values() to convert. Override checks go at top of createClient in db/drizzle/mod.ts.
- Driver test suite is entirely broken by pre-existing esbuild error in engine/actor-driver.ts (#hwsAckState private field). Cannot run any driver-file-system tests. This is NOT caused by dynamic actor changes.
- Fixture actors in actors/ directory (dbActorRaw.ts, dbActorDrizzle.ts) are already loaded as both static and dynamic actors via registry-loader.ts. No need to create new fixture actors for SQLite proxy testing.
- Dynamic SQLite proxy tests should use describe.skipIf(!driverTestConfig.isDynamic) to only run when the dynamic registry variant is active.
- Tests depending on optional peer deps (@secure-exec/typescript, secure-exec) must use describe.skipIf with require.resolve to skip gracefully.
- GlobalActorOptionsInput is a z.input<> type. Cannot use `interface extends` with it. Use intersection type instead.
- Dynamic runtime status lives in src/dynamic/runtime-status.ts. Use standalone transition functions (transitionToStarting, etc.) rather than class methods for cross-driver compatibility.
- coalesceDynamicStartup in startup-coalescing.ts orchestrates dynamic actor startup with coalescing (concurrent requests join existing promise) and generation tracking (stale completions discarded). Integrate it into driver startActor paths.
- When moving code into closures, capture narrowed TypeScript variables into local constants first to avoid losing type narrowing.
- Global-state.ts catch block wraps errors in plain Error, losing ActorError metadata. Use ActorError.isActorError() to preserve error codes through the error chain.
- For errors that must include error codes in client responses, set public: true and explicit statusCode on the ActorError. Use isDev() to conditionally include or sanitize details.

  - deconstructError in common/utils.ts is the centralized error-to-response converter. For public errors it includes message, group, code, metadata. For non-public errors it returns a generic "Internal error" message.
  - isDev() from utils/env-vars.ts is the standard way to check environment. Production errors should be sanitized to prevent leaking internal details.
  - ActorError.statusCode defaults to 500 for non-public and 400 for public errors. Dynamic startup errors need explicit statusCode = 503.
