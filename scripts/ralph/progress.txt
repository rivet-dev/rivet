## Codebase Patterns
- BARE schema files live in engine/sdks/schemas/{protocol-name}/v{N}.bare with no build step. They are plain text schemas.
- Use MARK comments (# MARK: Section) to organize sections in BARE schemas, matching runner-protocol conventions.
- The runner-protocol directory only has .bare files (no .ts or codegen in the same dir). TypeScript versioned handlers are separate.
- KV types in kv-channel-protocol mirror runner-protocol KV types but omit KvListRequest, KvDropRequest, and KvMetadata.
- TypeScript protocol packages live in engine/sdks/typescript/{protocol-name}/ with package.json, tsconfig.json, tsup.config.ts, and src/index.ts.
- The runner-protocol TS is auto-generated from BARE via build.rs; kv-channel-protocol is hand-written following the same read/write/encode/decode pattern.
- BARE void types in unions are represented as `null` values (no read/write call needed, just the tag byte).
- New pnpm workspace packages must be added to pnpm-workspace.yaml.
- The driver already has `kvDeleteRange(actorId, start, end)` on the actor driver interface. Wire it through `DatabaseProviderContext.kv` when adding KV range operations.
- `deleteRange` uses half-open `[start, end)` semantics everywhere (driver, file-system global-state, KV instance).
- For chunk key range bounds, use `getChunkKeyRangeEnd(fileTag)` which returns a 4-byte key `[PREFIX, VERSION, CHUNK_PREFIX, fileTag+1]` that is lexicographically past all 8-byte chunk keys for that fileTag.
- The sqlite-native crate is NOT in the root Cargo workspace. It has its own `[workspace]` table in Cargo.toml to prevent auto-discovery. Run `cargo check` from within `rivetkit-typescript/packages/sqlite-native/`.
- napi-rs v2 is used for the native addon. `crate-type = ["cdylib"]`, `napi-build` in build-dependencies, `napi`+`napi-derive` in dependencies.
- Platform npm packages under `npm/` follow naming: `@rivetkit/sqlite-native-{platform}` with `os`/`cpu` constraints.
- serde_bare `#[derive(Serialize, Deserialize)]` on enums produces byte-compatible BARE unions. Variant order must match schema tag order. Unit variants = void types, newtype variants = struct types.
- `tokio::spawn` requires `Send + 'static` on captured types. When using generic stream/sink types with `tokio::spawn`, add those bounds explicitly.
- `futures-util` is needed for `SinkExt`/`StreamExt` traits when working with `tokio-tungstenite` WebSocket streams (split, next, send).
- Reconnect backoff constants: initial=1000ms, max=30000ms, multiplier=2, jitter=0-25%. Matches `engine/sdks/typescript/runner/src/utils.ts`.
- libsqlite3-sys (bundled) exports `sqlite3_close` not `sqlite3_close_v2`. Use `sqlite3_close` for closing databases.
- Use `std::mem::zeroed()` to initialize `sqlite3_vfs` and `sqlite3_io_methods` structs, then set individual fields. This avoids needing to know the exact types of optional v2+/v3+ fields.
- File metadata encoding: 1 byte version (0x01) + 8 bytes u64 LE size = 9 bytes total. Must be byte-identical to WASM VFS vbare serialization.
- VFS callbacks are sync C functions. Bridge to async KV channel via `rt_handle.block_on()`. Must not be called from within a tokio async context.
