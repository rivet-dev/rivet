## Codebase Patterns
- Workflow compaction key helpers live in `engine/packages/depot/src/conveyer/keys.rs`; branch-local manifest/staging keys stay under `BR/{branch_id}/CMP`, while global pin/proof/dirty indexes use reserved partitions `0x70..=0x75`.
- PITR and workflow compaction limit constants live in `engine/packages/depot/src/conveyer/constants.rs` and are re-exported at the `depot` crate root.
- Workflow compaction persisted payload structs live in `engine/packages/depot/src/conveyer/types.rs`; expose vbare encode/decode helpers there and cover them from `engine/packages/depot/tests/`.
- Depot page reads prefer PIDX/DELTA, then latest branch SHARD at or below the read cap, then workflow `CMP/cold_shard` refs and legacy cold manifest layers outside the UDB transaction.
- Workflow compaction signals and durable workflow state live in `engine/packages/depot/src/workflows/compaction.rs`; expose vbare encode/decode helpers there and cover them from `engine/packages/depot/tests/`.
- Workflow compaction branch workflows use `DATABASE_BRANCH_ID_TAG` as their only stable unique tag; the manager stores companion workflow ids before entering its durable loop.
- Workflow compaction dirty admission lives in `engine/packages/depot/src/conveyer/commit.rs`; runtime callers inject a signaler into `Db` to create or target the unique manager workflow.
- Workflow compaction manager planning refreshes FDB in `RefreshManager` activity, records active jobs in durable manager state, and signals persistent companion workflows instead of spawning per-job workflows.
- Workflow hot compaction writes staged LTX blobs under `CMP/stage/{job_id}/hot_shard`; successful outputs stay as the manager's active hot job until a manager install story publishes or rejects them.
- Workflow hot install belongs in the DB manager: accept only the matching active job fingerprint, validate staged checksums, copy to reader-visible `SHARD`, advance `CMP/root`, and clear PIDX with `COMPARE_AND_CLEAR`.
- Unified DB history pin helpers live in `engine/packages/depot/src/conveyer/history_pin.rs`; workflow hot planning carries exact pinned txids in `HotJobInputRange.coverage_txids`.
- Workflow FDB reclaim is planned by the DB manager from current manifest, DB pins, PIDX, and SHARD coverage, then revalidated and deleted by the reclaimer with `COMPARE_AND_CLEAR`.
- Namespace fork proof facts are written from namespace/catalog mutation paths; workflow reclaim must resolve `NS_CHILD`/`NS_FORK_PIN` into exact `DB_PIN(kind=NamespaceFork)` records or retain history.
- Workflow cold compaction uploads through `ColdTier`, publishes `CMP/cold_shard` refs from the DB manager, and should schedule before reclaim so reclaim cannot delete the exact hot inputs cold publish revalidates.
- Workflow cold-object reclaim retires exact manager-provided `ColdShardRef` refs before S3 deletion, waits the grace window, marks `DeleteIssued`, deletes the exact object key, and leaves a completed retired record to block republish.
- Workflow branch lifecycle is fenced by `DatabaseBranchRecord.lifecycle_generation`; planned workflow jobs carry the base generation and reject stage, publish, or reclaim work after branch deletion or recreation.
- Workflow repair cleanup is reclaimer-owned: unmatched hot completions clear exact staged FDB rows, unmatched cold completions delete unreferenced S3 objects, and live cold refs with missing S3 or `DeleteIssued` retired records are logged and retained.
- Workflow force-compaction is manager-owned: send `ForceCompaction`, bypass only lag thresholds in `RefreshManager`, and wait on durable `force_compaction_results`.
